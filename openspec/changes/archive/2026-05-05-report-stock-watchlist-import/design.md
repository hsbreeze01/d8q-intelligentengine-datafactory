## Context

研报搜索页面当前通过单个输入框让用户手动输入股票名称/关键词，调用 `POST /api/report/stock`（接受 `keywords` 数组）搜索研报。用户跟踪多只股票时需要反复操作，效率低。

现有技术栈：
- 前端：单页 HTML（`templates/index.html`），纯 JS 无框架
- 后端：Factory（`app.py`）作为代理转发到 Infopublisher 的研报 API
- 研报 API 已支持 `keywords` 数组参数，无需后端改动

## Goals / Non-Goals

**Goals:**
- 支持用户上传 txt 文件导入股票名称列表
- 界面提供清晰的文件格式说明（每行一个股票名称、编码 UTF-8）
- 导入后批量搜索并合并展示所有股票的研报
- 支持文件内容预览和基本校验（空文件、行数过多等）

**Non-Goals:**
- 不做用户关注列表的持久化存储（本次仅做一次性导入搜索）
- 不做 CSV/Excel 等其他格式的支持（仅 txt）
- 不修改后端研报 API（现有接口已满足需求）

## Decisions

### 1. 纯前端解析 txt 文件（不经过后端上传）

**选择**: 使用浏览器 `FileReader API` 在前端直接读取 txt 文件内容，解析为关键词数组，调用现有 `/api/report/stock` 接口。

**理由**:
- 避免新增文件上传 API 和服务端存储逻辑
- txt 文件通常很小（几十行股票名称），前端解析无性能问题
- 现有 API 已支持 `keywords` 数组，无需后端改动

**替代方案**: 上传到后端解析 — 增加了不必要的 API 复杂度，且文件不含敏感信息。

### 2. UI 集成到现有研报搜索区域

**选择**: 在搜索输入框旁边增加"导入列表"按钮，点击后弹出文件选择 + 格式说明弹框。

**理由**: 保持与现有页面风格一致，用户不需要跳转页面。

### 3. 批量搜索策略：分批请求

**选择**: 当导入股票数量 > 10 时，分批调用 API（每批 10 个关键词），前端显示进度。

**理由**: 避免单次请求过大导致超时；研报 API 可能对 keywords 数组长度有限制。

## Risks / Trade-offs

- **[大量股票导致请求缓慢]** → 分批请求 + 进度条展示，每批 10 个关键词
- **[文件编码非 UTF-8]** → FileReader 默认 UTF-8，乱码时提示用户检查编码
- **[股票名称无效]** → API 返回空结果时正常显示"未找到"，不做前端校验（无法判断哪些名称有效）
