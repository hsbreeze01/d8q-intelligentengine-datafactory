# D8Q 高端用户需求规划 — 一级市场投资人场景

## 1. 目标用户画像

### 1.1 核心用户：一级市场投资经理/分析师

| 维度 | 描述 |
|------|------|
| 职位 | VC/PE 投资经理、行业分析师、投后管理 |
| 日常工作 | 看项目、做行研、写周报、跟踪 portfolio 公司动态 |
| 信息来源 | 36氪、IT桔子、财联社、研报、微信群、行业会议 |
| 痛点 | 信息碎片化、周报写作耗时、行业趋势判断依赖经验 |
| 决策周期 | 长（月级），但信息需求是日级 |
| 付费决策 | 个人付费或公司报销，对价格不敏感，对质量敏感 |

### 1.2 扩展用户

| 用户类型 | 与核心用户的关系 | 需求差异 |
|----------|------------------|----------|
| FA（财务顾问） | 需要行业热度数据说服 LP/买方 | 更关注市场情绪和估值趋势 |
| 产业投资人（CVC） | 关注特定赛道的技术演进 | 需要更深的技术维度 |
| 财经自媒体 | 需要高质量内容素材 | 关注内容可发布性和独特视角 |
| LP（有限合伙人） | 需要了解所投基金关注的赛道 | 更宏观，关注大趋势 |

## 2. 核心场景与功能需求

### 场景一：赛道热度追踪（P0 — 最高优先级）

**用户故事：** 作为一级市场投资经理，我需要实时了解我关注的赛道（如具身智能、AI Agent、合成生物）的市场热度变化，以便判断投资时机和方向。

**当前能力：** 已有按主题采集资讯的能力（subject 字段），但缺乏热度量化和趋势可视化。

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 赛道订阅管理 | 用户自定义关注赛道（关键词组合），不限数量 | P0 |
| 热度指数 | 基于资讯量、研报提及频次、政策信号的综合热度评分 | P0 |
| 热度趋势图 | 按日/周/月展示赛道热度变化曲线 | P0 |
| 热度异动提醒 | 某赛道资讯量突增时推送通知 | P1 |
| 赛道对比 | 多个赛道热度横向对比 | P1 |

**实现路径：**
- 基于现有 `financial_news` 表的 `subject` + `publish_time` 做聚合统计
- 新增热度评分算法（加权：资讯量 × 0.4 + 研报提及 × 0.3 + 政策信号 × 0.3）
- 前端新增赛道 Dashboard 页面

---

### 场景二：定制化周报生成（P0 — 杀手级场景）

**用户故事：** 作为投资经理，我每周需要向投委会/LP 提交行业周报，目前手动整理需要 3-5 小时。我希望平台能自动生成初稿，我只需审阅修改。

**当前能力：** 已有 LLM 研究报告生成（`research_report` 风格），但格式固定、不可定制。

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 周报模板管理 | 用户自定义周报结构（章节、小标题、数据维度） | P0 |
| 一键生成周报 | 选择赛道 + 时间范围 → AI 生成结构化周报 | P0 |
| 周报编辑器 | 在线编辑生成的周报，支持修改、补充、删除段落 | P0 |
| 多格式导出 | 导出为 Markdown / PDF / Word | P1 |
| 周报历史 | 查看和对比历史周报 | P1 |
| 定时自动生成 | 每周五下午自动生成并推送 | P2 |

**周报模板示例：**
```
一、本周赛道概览
  - 热度变化（对比上周）
  - 关键事件摘要（3-5 条）

二、政策与监管动态
  - 国内政策
  - 海外政策

三、融资与并购
  - 本周重要融资事件
  - 并购/IPO 动态

四、技术与产品进展
  - 重要产品发布
  - 技术突破

五、投资观点
  - 机构观点汇总
  - AI 综合研判

六、下周关注
  - 即将发布的数据/事件
  - 建议关注的方向
```

**实现路径：**
- 扩展 `llm_creator.py`，新增 `weekly_report` 风格
- 模板存储在 `data/report_templates.json`，用户可自定义
- 新增周报编辑页面（富文本编辑器）
- 导出功能使用 `python-docx` / `weasyprint`

---

### 场景三：政策信号监测（P0）

**用户故事：** 一级市场投资决策高度依赖政策风向。我需要第一时间知道与我关注赛道相关的政策发布、监管动态、领导讲话。

**当前能力：** 资讯采集中包含政策类内容，但未做专门分类和提取。

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 政策分类标签 | AI 自动识别资讯中的政策/监管类内容 | P0 |
| 政策影响分析 | AI 解读政策对特定赛道的利好/利空影响 | P0 |
| 政策时间线 | 按赛道展示政策演进脉络 | P1 |
| 政策预警 | 重大政策发布时即时推送 | P1 |

**实现路径：**
- 在资讯入库时增加 LLM 分类步骤（category 字段扩展：政策/融资/产品/技术/市场）
- 政策类资讯触发二次 LLM 调用，生成影响分析
- 新增数据源：国务院、发改委、工信部等政府网站

---

### 场景四：投融资事件追踪（P1）

**用户故事：** 我需要知道我关注赛道里谁在融资、谁在投、估值多少，以便判断市场热度和竞争格局。

**当前能力：** 无。需要新增数据源。

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 投融资事件采集 | 从 36氪、IT桔子、企查查等源采集融资事件 | P1 |
| 事件结构化 | 提取：公司、轮次、金额、投资方、赛道 | P1 |
| 赛道融资统计 | 按赛道汇总融资笔数、总金额、平均估值 | P1 |
| 投资方图谱 | 某赛道的活跃投资机构排名 | P2 |
| 一二级联动 | 融资事件与二级市场相关标的关联 | P2 |

**实现路径：**
- 新增爬虫适配器（36氪、IT桔子）
- 新增 `investment_events` 数据表
- LLM 辅助结构化提取

---

### 场景五：竞品/Portfolio 公司监控（P1）

**用户故事：** 我需要持续跟踪已投公司和竞品的动态（新闻、产品发布、人事变动），不想每天手动搜索。

**当前能力：** 资讯采集按主题（subject）聚合，但不支持按公司名聚合。

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 公司关注列表 | 用户添加关注的公司名称 | P1 |
| 公司动态聚合 | 自动匹配资讯中提及的关注公司 | P1 |
| 公司动态摘要 | 每日/每周生成关注公司的动态汇总 | P1 |
| 竞品对比 | 同赛道多家公司的动态并排展示 | P2 |

**实现路径：**
- 资讯入库时增加实体识别（NER），提取公司名
- 新增 `company_mentions` 关联表
- 用户端新增"我的关注"管理页面

---

### 场景六：推送与分发（P1）

**用户故事：** 我不会每天打开网页看，需要重要信息主动推送到我常用的渠道。

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 微信服务号推送 | 每日早报 + 异动提醒推送到微信 | P1 |
| 邮件订阅 | 周报自动发送到邮箱 | P1 |
| 企业微信/飞书 webhook | 推送到团队工作群 | P2 |
| 知识星球同步 | 内容自动同步到知识星球 | P2 |

**实现路径：**
- 微信服务号：申请服务号 + 模板消息 API
- 邮件：SMTP 发送（成本最低，优先实现）
- Webhook：HTTP POST，最简单

## 3. 优先级排期

### Phase 1：种子用户验证（第 1-4 周）

目标：让 10-20 个一级市场朋友开始使用，收集反馈。

| 功能 | 工作量 | 说明 |
|------|--------|------|
| 赛道订阅管理 | 3 天 | 用户自定义关注赛道 |
| 热度指数 + 趋势图 | 5 天 | 基于现有数据做聚合统计 + 前端图表 |
| 周报模板 v1 | 5 天 | 固定模板，一键生成，Markdown 导出 |
| 政策分类标签 | 3 天 | 入库时 LLM 分类 |
| 邮件推送 | 2 天 | SMTP 发送周报到邮箱 |
| **合计** | **~3 周** | |

交付物：可用的赛道追踪 + 周报生成 + 邮件推送，邀请种子用户试用。

### Phase 2：功能深化（第 5-10 周）

目标：根据种子用户反馈迭代，开始收费验证。

| 功能 | 工作量 | 说明 |
|------|--------|------|
| 周报编辑器 | 5 天 | 在线编辑 + PDF/Word 导出 |
| 政策影响分析 | 3 天 | 政策类资讯的 AI 解读 |
| 热度异动提醒 | 3 天 | 阈值触发 + 邮件/微信通知 |
| 投融资事件采集 v1 | 5 天 | 36氪数据源 + 结构化提取 |
| 公司关注列表 + 动态聚合 | 5 天 | NER + 关联匹配 |
| 微信服务号推送 | 5 天 | 服务号申请 + 模板消息 |
| **合计** | **~4 周** | |

交付物：完整的专业版功能，开始向种子用户收费（299 元/月）。

### Phase 3：规模化准备（第 11-16 周）

目标：产品稳定后，开始 C 端获客。

| 功能 | 工作量 | 说明 |
|------|--------|------|
| 用户注册 + 会员体系 | 5 天 | 注册、会员等级、到期管理 |
| 支付接入 | 5 天 | 微信支付/支付宝 |
| 移动端 H5 | 10 天 | 响应式页面或小程序 |
| 多平台发布 | 5 天 | 公众号、知识星球 |
| 基础版/进阶版产品化 | 5 天 | 功能分级、权限控制 |
| **合计** | **~5 周** | |

交付物：完整的 C 端产品，小红书引流 → 注册 → 付费闭环。

## 4. 关键指标

### Phase 1 验证指标

| 指标 | 目标 |
|------|------|
| 种子用户数 | ≥ 10 人 |
| 周活跃率 | ≥ 60%（每周至少登录 1 次） |
| 周报使用率 | ≥ 50%（每周至少生成 1 次） |
| NPS（净推荐值） | ≥ 30 |
| 用户反馈收集 | 每用户每周 ≥ 1 条有效反馈 |

### Phase 2 商业验证指标

| 指标 | 目标 |
|------|------|
| 付费转化率 | 种子用户中 ≥ 30% 愿意付费 |
| 月付费用户 | ≥ 5 人 |
| MRR（月经常性收入） | ≥ 1500 元 |
| 用户留存（月） | ≥ 80% |

## 5. 技术架构扩展方向

```
当前架构:
Agent(采集) → SQLite → Factory(创作+调度) → InfoPublisher(发布)

扩展架构:
                                    ┌→ 邮件推送
Agent(采集) → PostgreSQL → Factory ─┼→ 微信推送
  ↑                          ↓      ├→ InfoPublisher(小红书)
  ├ 36氪爬虫                 ↓      └→ 公众号/知识星球
  ├ 政策源                 周报引擎
  └ 公告源                   ↓
                          用户端(H5/小程序)
                             ↓
                          支付系统
```

关键技术决策：
- **数据库**：SQLite → PostgreSQL（多用户并发、全文检索）
- **推送**：新增消息队列（Redis）处理异步推送
- **NER**：使用 LLM 做实体识别，不引入额外 NLP 模型
- **前端**：现有模板渲染保持，移动端用独立 H5
